# MCP/Skill 项目评估归档 Agent — 技术规格说明书

## 1. 项目概述

### 1.1 项目定位

一个基于 **lark-cli skill** 的评估归档 Agent，自动解析 Git 仓库链接，调用 codegraph + LLM 分析项目信息，按统一维度分类、打标、评分后批量写入飞书多维表格，并输出汇总报告。

### 1.2 核心目标

- 解决 MCP 和 Skill 市场泛滥、质量参差不齐的问题
- 建立统一的评估维度和分类体系，便于按需筛选
- 纯入库工具，不做推荐引擎，筛选和使用由用户在飞书表格中自行完成

## 2. 整体流程

```
用户输入（单个 URL / 文本段落含多个 URL）
    │
    ▼
┌───────────────────────────────────────────────┐
│ 阶段一：链接提取 + 本地去重                     │
│ 输入 → 提取 GitHub 仓库 URL                    │
│       → 排除非仓库页面（_NON_REPO_OWNERS）     │
│       → 标准化（owner/repo 指纹去重）           │
│       → 查本地 imported.txt 已入库清单          │
│       → 已入库的跳过，不进后续分析              │
│       → 并发控制（最多 3 条并行）               │
└──────────────────┬────────────────────────────┘
                   │ (仅未入库的新项目)
                   ▼
┌───────────────────────────────────────────────┐
│ 阶段二：逐条数据采集                           │
│ 每条独立执行：                                 │
│   1. 获取 README → raw.githubusercontent.com   │
│   2. 获取 Stars → api.github.com (JSON API)    │
│   无需 git clone，无需配置文件                  │
└──────────────────┬────────────────────────────┘
                   │ (所有有效项目)
                   ▼
┌───────────────────────────────────────────────┐
│ 阶段三：一次 LLM 分析                          │
│ 一次 Prompt 同时输出：                         │
│ ├─ 一句话简介                                  │
│ ├─ 项目类型（MCP / Skill / Agent工具 / 项目）  │
│ ├─ 运行形式（MCP-stdio/MCP-SSE/Skill/不适用） │
│ ├─ 给谁用（Agent 调用 / 本地运行 / 两者皆可）  │
│ ├─ 功能领域（单选主分类）                       │
│ ├─ 能力标签（多选自由标签）                     │
│ ├─ 核心亮点                                    │
│ ├─ 文档评分（1-10）                            │
│ └─ 功能评分（1-10）                            │
│                                                │
│ 社区评分：本地按 Stars 换算，不占 LLM 调用      │
└──────────────────┬────────────────────────────┘
                   ▼
┌───────────────────────────────────────────────┐
│ 阶段四：飞书入库 + 本地清单更新                  │
│ ├─ 写入飞书多维表格（所有项目，不分类型）       │
│ ├─ 成功后追加 owner/repo 到 imported.txt       │
│ └─ 单条失败不中断，最多重试 3 次               │
└──────────────────┬────────────────────────────┘
                   ▼
┌───────────────────────────────────────────────┐
│ 阶段五：输出汇总报告                            │
│ ├─ 总输入 | 已入库跳过 | 成功入库 | 失败       │
│ ├─ 按类型分布：MCP / Skill / Agent工具 / 项目 │
│ └─ 失败明细（链接 + 原因）                     │
└───────────────────────────────────────────────┘
```

## 3. 阶段一：链接提取与去重

### 3.1 输入格式

支持两种输入形式：

1. **直接 URL**：用户直接发送一个 GitHub 仓库链接
2. **文本段落**：用户发送一段文字，由 LLM 自动提取其中的 GitHub 仓库 URL

### 3.2 链接提取规则

- 匹配模式：`https://github.com/{owner}/{repo}` 及 `git@github.com:{owner}/{repo}.git`
- 忽略非 GitHub 链接
- 忽略 GitHub 非仓库页面（通过 `_NON_REPO_OWNERS` 黑名单过滤 Profile、Settings、Notifications 等）
- 输出：标准化后的 `https://github.com/{owner}/{repo}` 格式

### 3.3 去重策略

去重分两层，都在阶段一完成：

#### 第一层：批次内去重

- 基于 `{owner}/{repo}` 指纹去重
- `https://github.com/A/B.git` 和 `git@github.com:A/B.git` 视为同一项目
- 同一批次输入中出现多次 → 只保留一次

#### 第二层：本地已入库清单比对

- 项目维护一个本地文本文件 `imported.txt`，每行一个已成功入库的 `owner/repo`
- 每次启动时逐行读入内存 `Set`，O(1) 查重
- 提取链接后，先查本地清单：
  - 已在清单中 → 标记为"已入库跳过"，不进后续分析管线
  - 不在清单中 → 进入阶段二分析
- 与飞书解耦：用户可在飞书表格中自由过滤/删除数据，不影响去重逻辑
- 写入飞书成功后，立即 `echo owner/repo >> imported.txt` 追加

#### imported.txt 格式

```
# 每行一个 owner/repo
owner/repo-a
owner/repo-b
owner/repo-c
```

## 4. 阶段二：逐条数据采集

### 4.1 数据采集路径

每条新链接（未入库 + 未在批次内去重）独立执行数据采集，全程通过 HTTP 请求访问两个公开 URL，无需 git clone，无需拉取配置文件。

```
┌─ 数据采集 ──────────────────────────────────┐
│                                               │
│  [A] HTTP GET → README.md                     │
│  ├─ https://raw.githubusercontent.com/        │
│  │  {owner}/{repo}/main/README.md             │
│  ├─ 失败则降级尝试 master 分支                │
│  └─ 获取原始 Markdown 内容                    │
│                                               │
│  [B] HTTP GET → GitHub API 取 Stars           │
│  ├─ https://api.github.com/repos/             │
│  │  {owner}/{repo}                            │
│  ├─ JSON 响应，解析 stargazers_count 字段     │
│  └─ 用于社区评分换算                          │
│                                               │
└───────────────────────────────────────────────┘
```

阶段二不做任何过滤——无论项目是 MCP、Skill、Agent 工具还是普通项目，都完整采集数据后进入阶段三 LLM 分析。**"是否跳过"的逻辑已被移除**，类型的区分由阶段三的 LLM 统一完成。

### 4.2 数据来源总览

| 来源 | 获取方式 | 说明 |
|------|---------|------|
| README.md | HTTP GET → raw.githubusercontent.com | 核心语义来源（项目功能、分类依据） |
| Stars 数 | HTTP GET → api.github.com (REST API) | 社区评分依据（JSON 解析 `stargazers_count`） |

## 5. 阶段三：一次 LLM 分析

### 5.1 核心思路

将原设计中的多次 LLM 调用合并为一次 Prompt，同时输出所有评估结果。
社区评分不占用 LLM 调用，在本地按 Stars 换算表直接计算。

### 5.2 一次 Prompt 的输出字段

```
├─ 一句话简介        → 项目是干什么的（20字内）
├─ 项目类型          → MCP / Skill / Agent工具 / 项目
├─ 运行形式          → MCP-stdio / MCP-SSE / Skill / 不适用
├─ 给谁用            → Agent 调用 / 本地手动运行 / 两者皆可
├─ 功能领域          → 单选主分类（从分类体系中选择）
├─ 能力标签          → 多选自由标签（描述具体能力）
├─ 核心亮点          → 区别于同类项目的独特优势
├─ 文档评分（1-10）  → LLM 评估 README 完善度
└─ 功能评分（1-10）  → LLM 评估功能完整度
```

Prompt 输入数据：项目 README 内容 + Stars 数

> "项目类型"替代了原设计的"是否 MCP/Skill 二值判定"和"是否通用型"两个字段，将分类从"是/否"变成"四选一"，覆盖了 MCP、Skill、Agent工具（非标准 MCP 但供 Agent 使用的工具）和普通项目（开源库、CLI 工具等）。

### 5.3 分类体系

#### 维度一：项目类型（四选一）

```
MCP         → 实现 MCP 协议标准的服务器
Skill       → Lark CLI 可加载的 skill
Agent工具   → 供 AI Agent 使用但非标准 MCP 的工具
项目        → 普通开源项目（库、CLI 工具等）
```

#### 维度二：运行形式

```
MCP-stdio    → 本地命令行启动的 MCP 服务器
MCP-HTTP-SSE → 远程 HTTP 服务的 MCP 服务器
Skill        → Lark CLI 可加载的 skill
不适用       → 非 MCP/Skill 项目，此维度不适用
```

#### 维度三：功能领域（单选主分类）

基于用户的实际用途设计：

```
├─ 🎨 PPT/演示文稿生成
├─ 📚 学习资料/课程制作
├─ 🎵 音频生成/处理
├─ 🎬 AI 视频生成/编辑
├─ ✍️ AI 写作辅助（含小红书、公众号文案）
├─ 🔒 代码安全防御
├─ 🧪 代码测试/调试
├─ 📦 项目管理/CI
├─ 🏗️ 架构/设计
├─ 📊 数据分析与可视化
├─ 🤖 AI 模型交互/编排
├─ 🔧 通用工具（文件/网络/搜索等）
└─ 其他（需人工标注）
```

#### 维度四：能力标签（多选自由标签）

描述具体能力，如：
```
["AI生成PPT", "支持自定义模板", "导出为PDF"]
["代码审查", "注入检测", "性能扫描"]
["文字转语音", "多语言TTS"]
```

#### 维度五：使用方

```
Agent 调用      → 供 AI Agent 直接调用的工具
本地手动运行    → 需用户本地安装运行的 CLI 工具
两者皆可       → 既可作为工具被 Agent 调用，也可本地运行
```

### 5.4 评分模型

#### 三个评分维度

| 维度 | 分值范围 | 计算方式 |
|------|---------|---------|
| ⭐ 社区评分 | 1-10 | **本地换算**：Stars 分档换算，不占 LLM |
| 📝 文档评分 | 1-10 | LLM 评估 README 完善度（在合并 Prompt 中一起输出） |
| 🔧 功能评分 | 1-10 | LLM 评估功能完整度（在合并 Prompt 中一起输出） |

#### 社区评分换算表（本地计算）

| Stars 范围 | 分数 |
|-----------|------|
| 0 - 10 | 1 |
| 11 - 100 | 2 |
| 101 - 500 | 3 |
| 501 - 1000 | 4 |
| 1001 - 5000 | 5 |
| 5001 - 10000 | 6 |
| 10001 - 30000 | 7 |
| 30001 - 100000 | 8 |
| 100001+ | 9-10（根据影响力微调） |

#### 综合评分

```
综合评分 = 社区评分 + 文档评分 + 功能评分
（总分 30 分，无权重，直接求和）
```

#### 文档评分评估维度（LLM 判断）

| 维度 | 权重占比 |
|------|---------|
| 有项目简介 | 20% |
| 有安装/配置说明 | 20% |
| 有使用示例 | 25% |
| 有 API/配置参数说明 | 20% |
| 有贡献指南/FAQ/常见问题 | 15% |

#### 功能评分评估维度（LLM 判断）

| 维度 | 权重占比 |
|------|---------|
| 有明确的 MCP 工具/命令定义 | 30% |
| 有完整的代码结构 | 25% |
| 有测试代码 | 15% |
| 有错误处理机制 | 15% |
| 有 CI/CD 配置 | 15% |

## 6. 阶段四：飞书入库 + 本地清单更新

### 6.1 飞书表格字段 Schema

| 字段名 | 字段类型 | 必填 | 说明 |
|--------|---------|------|------|
| `项目名称` | 文本 | 是 | GitHub repo 名称 |
| `Git 地址` | 链接 | 是 | 标准化后的 GitHub URL |
| `项目类型` | 单选 | 是 | MCP / Skill / Agent工具 / 项目 |
| `开发语言` | 文本 | 否 | 从 README 或 GitHub 信息提取 |
| `项目描述` | 文本 | 否 | README 一句话简介 |
| `运行形式` | 单选 | 否 | MCP-stdio / MCP-SSE / Skill / 不适用 |
| `给谁用` | 单选 | 否 | Agent 调用 / 本地运行 / 两者皆可 |
| `功能领域` | 单选 | 是 | 主分类（见 5.3 分类体系） |
| `能力标签` | 多选 | 否 | 自由标签描述具体能力 |
| `核心亮点` | 文本 | 否 | 区别于同类项目的独特优势 |
| `社区评分` | 数字 | 是 | 1-10 |
| `文档评分` | 数字 | 是 | 1-10 |
| `功能评分` | 数字 | 是 | 1-10 |
| `综合评分` | 数字 | 是 | 三项之和（3-30） |
| `评估日期` | 日期 | 是 | 入库时间 |
| `状态` | 单选 | 否 | 待评估 / 已入库 / 选中使用 |

### 6.2 飞书 API 调用

通过 lark-cli 的 `base` 命令操作飞书多维表格：

```
lark-cli base +create-record <base_token> <table_id> --fields <json>
```

### 6.3 本地清单更新

写入飞书成功后，立即更新本地 imported.txt：

```
每次写入成功 → echo owner/repo >> imported.txt
             → 一行一个，追加写入
```

imported.txt 格式示例：
```
# 每行一个 owner/repo
owner/repo-a
owner/repo-b
owner/repo-c
```

### 6.4 错误处理

- 单条写入失败 → 记录错误原因 → 继续处理下一条
- 重试策略：最多重试 3 次，间隔 1s
- 网络超时：15s 超时阈值
- 飞书写入失败不影响本地清单：只有写入成功才更新 imported.txt

## 8. 技术架构

### 8.1 技术栈

| 组件 | 版本 | 用途 |
|------|------|------|
| lark-cli | latest | Agent 载体 + 飞书多维表格写入 |
| web-access skill | 2.5.1 | WebFetch 获取 GitHub 仓库内容（README、配置文件、元数据） |
| LLM (Claude) | 当前模型 | 语义理解、分类、评分（一次 Prompt 合并输出） |
| Python | >=3.10 | 辅助脚本（链接提取、数据采集、字段映射、报告生成） |
| requests | >=2.28 | HTTP 客户端库（GitHub API 调用） |

### 8.2 辅助脚本模块

项目提供 `assets/` 目录下的 Python 辅助脚本，用于各阶段的机械性操作：

| 模块 | 职责 | 对应阶段 |
|------|------|---------|
| `extractor.py` | URL 正则匹配、标准化、指纹去重、imported.txt 比对 | 阶段一 |
| `collector.py` | HTTP 请求获取 README、GitHub API 获取 Stars | 阶段二 |
| `analyzer.py` | LLM Prompt 模板、`AnalysisResult` 数据模型、JSON 解析 | 阶段三 |
| `feishu_writer.py` | lark-cli base 命令封装、重试逻辑 | 阶段四 |
| `tracker.py` | imported.txt 读写、初始化 | 阶段二/四 |
| `reporter.py` | `ReportItem`/`Report` 数据模型、格式化输出 | 阶段五 |
| `main.py` | 命令行入口，编排 5 阶段流程 | 全流程 |

### 8.3 组件交互关系

```
┌──────────────┐
│  用户输入     │  (URL / 文本)
└──────┬───────┘
       │
┌──────▼──────────────────────────────────────┐
│  lark-cli skill（batch_link_import Agent）   │
│                                              │
│  ├─ SKILL.md（主指令）                       │
│  ├─ assets/extractor.py（链接提取+去重）     │
│  ├─ assets/collector.py（数据采集）          │
│  ├─ assets/analyzer.py（LLM分析框架）        │
│  ├─ assets/feishu_writer.py（飞书写入）      │
│  ├─ assets/tracker.py（清单维护）            │
│  └─ assets/reporter.py（报告生成）            │
│                                              │
│  1. 调用 extractor → 链接提取 + 去重        │
│  2. 调用 collector → 采集 README + Stars    │
│  3. 调用 analyzer → 构建 Prompt 给 LLM      │
│  4. 调用 feishu_writer → 写入飞书            │
│  5. 调用 tracker → 更新 imported.txt         │
│  6. 调用 reporter → 输出汇总报告             │
└──────┬──────────────────────────────────────┘
       │
       ├──▶ web-access-2.5.1 (备选数据采集)
       │
       └──▶ lark-base skill (写入多维表格)
```

### 8.3 并发控制

- 最大并发数：3
- 控制策略：信号量模式，完成一个再调度下一个

## 9. 异常处理矩阵

| 异常场景 | 处理方式 | 是否中断 |
|---------|---------|---------|
| 链接非 GitHub 仓库 | 跳过，记录原因 | 否 |
| 仓库不存在/404 | 跳过，记录原因 | 否 |
| WebFetch 获取 README 或 Stars 失败 | 跳过，记录原因 | 否 |
| LLM 无法确认分类 | 标记为"待人工确认" | 否 |
| 飞书写入失败 | 重试 3 次，失败则记录 | 否 |
| 输入为空（无有效链接） | 提示用户，终止 | 是 |

## 10. 功能领域分类矩阵（决策树）

判定项目属于哪个功能领域，使用以下逻辑：

```
输入：项目描述 + README 内容 + 能力标签

问：这个 MCP/Skill 的主要用途是什么？

PPT/演示文稿类
  └─ README 含 "slide"、"presentation"、"ppt"、"演示"...

写作辅助类
  └─ README 含 "write"、"article"、"blog"、"文案"、"小红书"、"公众号"...

音视频类
  └─ README 含 "audio"、"video"、"TTS"、"speech"、"transcribe"...

代码安全类
  └─ README 含 "security"、"vulnerability"、"injection"、"audit"...

通用工具类
  └─ 不具备上述特征 → 默认分类

最终由 LLM 综合判断
```

## 11. 假设与约束

### 11.1 假设
- 目标项目全部托管在 GitHub 公开仓库
- 目标项目需具备基本的 README 文档
- 运行环境能访问 GitHub 和飞书 API

### 11.2 约束
- 不评估非开源项目
- 不评估非 Git 托管项目
- 不集成推荐引擎
- 不做定期数据更新/保鲜
- 不做能力冲突自动检测

### 13.2 开发过程中的关键决策

| 决策 | 方案 | 原因 |
|------|------|------|
| Stars 获取方式 | GitHub REST API (`api.github.com`) | 原生 GitHub 页面需 JS 渲染，API 返回结构化 JSON 更可靠 |
| 非仓库页面过滤 | `_NON_REPO_OWNERS` 黑名单 | 避免 `settings/profile`、`notifications` 等被误认为仓库 |
| 大小写处理 | 去重时大小写不敏感（全部 lowercase） | GitHub URL 的 owner/repo 在用途上大小写不敏感 |
| 重试策略 | 最多 3 次，间隔 1s | 应对临时网络故障 |
| 报告格式 | 纯文本控制台输出 | 在 Agent 工作流中可直接输出给用户 |

### 13.3 边界条件处理

| 场景 | 处理方式 | 验证结果 |
|------|---------|---------|
| 空输入/无有效链接 | 返回空列表，提示用户 | ✅ |
| 非仓库 GitHub 页面 | `settings/profile`、`notifications` 等被黑名单过滤 | ✅ |
| 重复链接（同格式 3 次） | 批次内去重为 1 次 | ✅ |
| 混合格式重复（HTTPS + SSH + .git） | 指纹去重为 1 次 | ✅ |
| 不存在的仓库 | 404 错误，记录失败原因 | ✅ |
| Stars 评分边界值 | 0~500K+ 共 10 个档位全部正确映射 | ✅ |
| lark-cli 不可用 | `feishu_writer` 优雅返回 False，不崩溃 | ✅ |
| README 在 master 分支 | 降级尝试 master 分支 | ✅ |
